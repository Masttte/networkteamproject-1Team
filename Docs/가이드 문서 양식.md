## 1. 프로젝트 개요

### 기본 정보
- **프로젝트명**: 미정
- **장르**: 비대칭 멀티플레이 생존 호러 (마피아 메커니즘)
- **컨셉**: 괴물이 갇혀있는 낯선 곳에 떨어진 생존자들. 그중 한 명은 괴물을 풀어주려는 마피아다. 생존자는 미션을 완수하며 마피아를 색출해야 하고, 마피아는 정체를 숨긴 채 괴물을 풀어 생존자를 제거한다.
- **시점**: 1인칭
- **플레이어 수**: 4인 (생존자 3 / 마피아 1)
- **플랫폼**: Windows Standalone
- **개발 기간**: 2026-04-24 ~ 2026-05-18 (약 3주)
- **팀 구성**: 5인
- **개발 환경**: Unity6 / C# / URP 3D / NGO + Relay

### 프로젝트 목표
- NGO를 사용한 넷코드 멀티플레이 구현
- 호스트-클라이언트 권한 분리 기반의 네트워크 구조 설계 경험

1. 생존자와 마피아간 몬스터를 바라보는 시야 차이
2. 생존자의 마피아 색출 및 몬스터 회피 긴장감
3. 마피아가 감시를 피해 괴물을 풀어주는 긴박함

### 승리 / 종료 조건
- 생존자 그룹은 미션을 완수하고 탈출하거나, 마피아를 찾아 처치한다
- 마피아는 괴물을 이용해 생존자 그룹을 전원 제거한다

## 2. 팀 협업 규칙

### 2.1 그라운드 룰

#### 작업 영역 준수
- 각자 작업 중인 영역을 침범하지 않는다
- 공동 사용 가능성이 높은 에셋을 변경하려면 팀원에게 알린다
- 타인 작업 영역은 무통보 수정 금지
- 작업 필요 시 복사본 생성 → 컨펌 후 적용
- 구현 완료 시 프리팹화하여 공유

#### 씬 관리 원칙
- **MainScene 관리 담당**: 
- **개인 테스트 씬**: `Test_이름` 형식으로 각자 폴더에서 작업
- **프리팹화 필수**: 작업물은 반드시 프리팹으로 제출
- 같은 씬을 두 명이 동시에 작업하지 않으며, 필요 시 복사 후 병합
- 병합은 프리팹 및 코드 단위로 진행
- 씬 수정 시 사전 통보 필수

#### 빌드 및 공유
- **빌드 주기**: 
- **PR 주기**: 
- 장시간 막힐 시 즉시 공유
- 매일 진행 상황 공유

#### 복구 방법
- 오류 시 Git Checkout으로 파일 되돌리기
- 씬이면 메타파일까지 함께 복구 (매우 중요)

### 2.2 Git 워크플로우

#### 브랜치 전략
1. **`main`**: 최종 빌드 및 배포용 (직접 푸시 금지)
2. **`develop`**: 개발 통합용 (모든 PR은 여기로)
3. **`feat/기능명_작성자`**: 개인 작업 브랜치

**브랜치 네이밍 규칙**: `접두사/기능_작성자` (예: `feat/player-move_PEY`)

**작업 순서**:
```
develop에서 브랜치 생성 
→ 작업 
→ develop으로 PR 
→ Merge 후 브랜치 삭제
```

- 사용하지 않는 오래된 브랜치는 삭제하여 혼동 방지

#### 커밋 컨벤션
- **커밋 전 필수 체크**: 콘솔 에러 0개, 테스트 플레이 완료
- 작은 변경의 경우 제목으로 설명하고 본문 작성은 생략 가능

**커밋 메시지 prefix**:
- `feat:` - 새로운 기능 추가
- `fix:` - 버그 수정
- `docs:` - 문서 수정
- `refactor:` - 코드 리팩토링
- `test:` - 테스트 코드
- `chore:` - 빌드, 설정 변경

참고: [Udacity Git Commit Style Guide](https://udacity.github.io/git-styleguide/)

#### PR 규칙
- 병합 전 콘솔 로그 확인 및 크리티컬 버그 테스트
- main 직접 푸시 지양, 브랜치 병합 기본
- 병합 담당자: 
- 간단한 수정은 개인 판단 가능하나 팀원에게 반드시 공유

### 2.3 소통 규칙

#### 필수 통보 사항
- 머지 완료 시 디스코드 통보
- 외부 플러그인 추가 시 통보
- 일정 문제 발생 시 미리 통보
- 장시간 막힐 시 즉시 공유

#### 문서 관리
- 노션/깃허브 문서: 
- **작업 노트**: 각자 기능 브랜치에서 자유롭게 작성, 깃허브 `Docs/` 폴더에 함께 커밋 권장
- **가이드/역할 분담 문서**: 팀장 관리, 수정 요청 시 별도 브랜치로 작업

#### 작업 관리 및 피드백
- 작업을 세부 기능 단위로 분리하여 Notion에 올리고 진행 상황 공유
- 작업 문서를 작성하여 깃허브 `Docs/` 폴더에 함께 커밋 권장

#### 태도 및 작업 관련
- 작업 시간 중 집중
- 각자 컨디션 관리

### 2.4 에셋 관리

#### 유료 / 외부 에셋
- **GitHub 업로드 금지** (라이선스 이슈)
- 모든 외부 에셋은 `Imports` 폴더에 배치 후 `.gitignore` 처리
- 공유 경로: [Google Drive](https://drive.google.com/drive/folders/1izjsutvxADilHC09RIWbx_3Hiw-QuYHc)

#### 외부 플러그인
- 사용 전 라이선스 확인 및 팀장 승인 필수
- 해당 경로 `.gitignore` 처리
- 직접 제작한 플러그인은 작업 노트에 기록 후 업로드 가능

### 2.5 프리팹 관리 규칙
- 대분류는 폴더 트리 이름으로 분리
- 세부 네이밍은 `속성_이름_특이사항` 형태로 **최소 2가지 이상** 요소 포함
- 예: `Prefabs/Characters/Enemy/Orc_Boss`, `Prefabs/UI/Player_Hpbar_Blue`
- 공통 적용 프리팹은 원본 수정, 그 외는 복사본/Variant 사용

### 3.3 코드 컨벤션

#### 네이밍 규칙
```csharp
// 클래스/구조체/열거형: PascalCase
public class PlayerController { }

// 메서드: PascalCase
void MovePlayer() { }

// 변수/매개변수: camelCase
float moveSpeed;

// private 필드: _camelCase (선택)
private float _currentSpeed;

// 상수: UPPER_CASE
const float MAX_SPEED = 10f;

// 프로퍼티: PascalCase
public bool IsGrounded { get; private set; }

// 인터페이스: I 접두사
public interface IDamageable { }

// 추상클래스: Base 접미사
public abstract class PlatformBase { }
```

#### 주석 작성 원칙
```csharp
// ✅ 좋은 주석 - WHY를 설명
// 경사면에서 이동 방향을 바닥에 투영하여 자연스러운 이동 구현
moveDir = Vector3.ProjectOnPlane(moveDir, groundNormal);

// ❌ 나쁜 주석 - WHAT만 설명
// 벡터 계산
moveDir = Vector3.ProjectOnPlane(moveDir, groundNormal);
```

- 네이밍을 명확하게 하여 과한 주석을 피하고 가독성을 높인다
- 핵심 코드 및 설명이 필요한 부분만 주석

## 6. 개발 워크플로우

### 6.1 작업 시작 전
1. 최신 버전 Pull
2. 씬 확인 (다른 팀원 수정 여부)
3. 브랜치 생성 (`feat/기능명_작성자`)

### 6.2 작업 중
1. 자주 테스트
2. 콘솔 에러 즉시 해결
3. 변경사항 주석 작성

### 6.3 작업 완료 후
1. 씬 저장 (Ctrl+S)
2. 테스트 플레이
3. 커밋 + 푸시 (의미있는 커밋 메시지)
4. PR 생성 및 팀원 공유

### 6.4 긴급 상황 대처

#### 씬이 깨짐
1. 백업 씬으로 복구
2. Git에서 이전 버전 체크아웃 (메타파일 포함)
3. 팀원에게 도움 요청

#### 코드가 작동 안 함
1. 콘솔 에러 메시지 확인
2. Debug.Log / Logger로 변수값 출력
3. 코드 리뷰 요청
